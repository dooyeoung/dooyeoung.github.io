복잡한 분석 로직을 체계적으로 처리하는 데 강점을 가지고 있습니다. 컨설팅 상품으로 판매하던 개인별 리포트 분석을 제품화하기 위해 R 언어로 구현된 분석 시스템을 Python으로 완전히 재구현하고, 등급형부터 주관식까지 제품 범위를 대폭 확장하여 2개월 만에 성공적으로 출시했습니다.

분석 결과 검증을 위한 내부 도구까지 개발하여 QA 과정의 효율성을 크게 향상시켰습니다. 또한 데이터 30,000개 위계 데이터 처리 시스템을 속도 향상(30s→0.5s)시키고, AWS 인프라 효율화를 통해 연간 5,000만 원 비용 절감을 달성했습니다.

팀의 안전망 역할과 긍정적인 태도로 동료들로부터 높은 신뢰를 받았습니다. 장애 상황에서의 신속 대응과 추진력 있는 소통으로 팀 사기와 생산성을 높여왔습니다.

사용자가 더 빠르고 편리하게 업무를 처리할 수 있는 도구 개발에 집중하며, 목표 엑셀 업로드 속도를 17초에서 3초로 개선하는 등 대량 데이터 처리 경험을 보유하고 있습니다. 정합성과 재사용성을 함께 고려한 솔루션을 찾으며, 다양한 제약 조건 속에서도 안정적이고 확장 가능한 시스템을 설계합니다. 투명하고 따뜻한 소통을 통해 동료들과 시너지를 만들어내는 개발자입니다.

개인별 리포트 분석 기능 개발

기존 컨설팅 상품으로 제공되던 개인별 분석 리포트를 자동화된 제품으로 전환해야 했습니다. 오랜 개발 히스토리와 암묵적 정책이 포함된 R 코드를 Python으로 변환하는 과정에서 분석 로직의 정확성 보장과 다양한 리뷰 유형(등급형, 다중선택, 주관식)에 대응하는 확장 가능한 구조 설계가 필요했습니다.


복잡한 분석 로직을 체계적으로 재구현하고 확장 가능한 시스템을 구축했습니다.

  • R 코드와 SQL 쿼리를 기능 단위로 분석하여 Python 기반으로 완전 재구현
  • 등급형 분석에서 시작하여 다중선택, 주관식까지 분석 범위를 단계적으로 확장
  • RDS 기반 분석 대기열 구조 설계로 리소스 부담 최소화
  • 분석 결과 정적파일 생성 및 S3 저장 시스템으로 데이터 불변성 확보
  • 분석 과정 데이터를 시각적으로 확인할 수 있는 내부 검증 도구를 개발하여 QA팀과 동료들에게 제공

2개월 만에 MVP를 성공적으로 출시하여 고객에게 즉시 가치를 제공했으며, 이후 4개월간 약속된 기능을 지속적으로 확장했습니다. 복잡한 상태 흐름 관리와 대량 데이터 처리 경험을 통해 확장 가능한 분석 시스템 구축 역량을 확보했습니다. 개발한 검증 도구를 통해 수작업에 의존하던 QA 시간을 대폭 단축하고, 데이터 기반의 정밀한 검증 프로세스를 구축하여 제품의 신뢰도를 높였습니다.


레거시 시스템의 핵심 로직을 새로운 기술로 마이그레이션할 때는 단순 번역이 아닌 비즈니스 로직의 본질적 이해가 중요함을 깨달았습니다. 특히 분석 결과의 정확성이 고객의 인사 평가에 직결되는 상황에서 체계적인 검증 프로세스와 내부 도구 개발의 중요성을 배웠습니다.

AWS 인프라 비용 절감 TF 참여

기존 AWS 인프라에서 파편화되어 운영되던 ECS 서비스들로 인한 비효율적인 리소스 사용과 높은 운영 비용 문제가 발생했습니다. 각각 독립적으로 운영되던 서비스들과 분산된 비동기 처리 시스템으로 인해 인프라 관리 복잡도가 증가하고 불필요한 비용이 지속적으로 발생하고 있었습니다.


이러한 문제를 해결하기 위해 체계적인 인프라 통합 작업을 수행했습니다.

  • 파편화된 ECS 서비스들을 도메인별로 분석 및 그룹핑
  • 관련성이 높은 서비스들을 하나의 통합된 서비스로 재구성
  • celery 워커를 새롭게 구성하여 기존 분산된 비동기 서비스들을 통폐합
  • 중복되는 인프라 리소스 제거 및 전체 아키텍처 최적화

이러한 인프라 개선 작업의 결과로 상당한 비용 절감 효과를 달성했습니다. 작업 완료 시점을 기준으로 월 3,445달러(약 473만 원)의 비용을 절감했으며, 이를 연간으로 환산하면 약 41,340달러(약 5,642만 원)에 해당하는 막대한 비용 절감 효과를 얻었습니다.


이 프로젝트를 통해 도메인 중심의 서비스 설계와 통합의 중요성을 깊이 이해하게 되었습니다. 단순히 기능적 분리가 아닌 비즈니스 도메인을 기준으로 한 서비스 구성이 운영 효율성과 비용 측면에서 훨씬 효과적임을 확인했습니다. 또한 비동기 처리 시스템의 통합을 통해 인프라 아키텍처 최적화 방법론을 습득했으며, 이는 향후 유사한 프로젝트에서 활용할 수 있는 귀중한 경험이 되었습니다.

목표 제품 관리

대규모 조직의 고객이 새롭게 유입되면서 목표 제품 관리 시스템 전반에 걸쳐 사용자 효율을 저해하는 문제들이 산재해 있었습니다. 대량의 데이터를 엑셀로 업로드할 때 N+1 쿼리로 인한 심각한 성능 저하가 발생했고, 복잡한 승인 정책을 테스트할 명확한 도구가 없어 QA 효율이 떨어졌습니다. 또한, 사용자가 목표를 확인하고 업데이트하는 과정이 번거로워 접근성 개선이 시급했습니다.


제품의 완성도를 높이기 위해 여러 핵심 기능을 체계적으로 개선했습니다.

  • 승인권자 미리보기 내부 도구를 개발하여 복잡한 정책 논의와 테스트 과정을 효율화했습니다.
  • 엑셀 업로드 기능의 N+1 쿼리를 제거하고 코드를 리팩토링하여 데이터 처리 속도를 개선했습니다. (17초 → 3초)
  • 사용자가 슬랙에서 바로 목표를 체크인할 수 있는 기능을 구현하고, 모바일 간편 체크인 기능을 추가하여 사용 편의성을 극대화했습니다.
  • SQS와 Celery Beat를 도입하여 안정적인 스케줄 처리 기반을 마련했습니다.

일련의 개선 작업을 통해 목표 제품 관리 시스템의 성능, 테스트 효율성, 사용자 접근성을 모두 크게 향상시켰습니다. 사용자는 대량의 목표를 빠르고 편리하게 관리할 수 있게 되었고, 개발팀은 신규 기능을 더 빠르고 안정적으로 테스트하고 배포할 수 있는 환경을 갖추게 되었습니다.


단일 기능 개선을 넘어, 시스템 전반의 사용자 여정과 개발 프로세스를 함께 고려하는 것이 제품의 가치를 극대화하는 길임을 배웠습니다. 특히 QA를 위한 내부 도구 개발이 팀 전체의 생산성과 최종 제품의 품질에 얼마나 큰 영향을 미치는지 체감했으며, 작은 접근성 개선이 사용자 만족도를 크게 높일 수 있다는 점을 깨달았습니다.

다양한 PoC 작성 및 공유, 제품화

레몬베이스 목표 제품 사용 과정에서 지속적으로 발생하는 사용자 VOC와 실무진의 불편함을 개선해야 하는 상황이었습니다. 목표 생성부터 체크인까지의 전체 사용자 여정에서 비효율적인 부분들이 산재해 있었고, QA 과정에서도 반복적인 수작업으로 인한 생산성 저하가 지속되고 있었습니다. 또한 사용자들이 자주 요청하는 기능들이 제품 로드맵에 반영되지 못하고 있어 제품 완성도 향상을 위한 체계적인 개선이 필요했습니다.


아래 아이디어들을 실험하고 구성원들에게 공유하였습니다.

  • 목표 생성 시 조직도 기반 담당자/조직 선택 기능을 React로 PoC 구현
  • 삭제된 목표 복구, 목표 이동 등 반복 VOC를 반영한 기능 PoC 개발
  • 목표 임시저장, 체크인 시 파일 첨부 등 실사용자 니즈 기능 PoC 구현
  • React Flow를 활용한 전체 목표 Tree view PoC 개발
  • 목표 체크인 현황 파악을 위한 대시보드 PoC 구현
  • 테스트 환경 구성을 위한 구성원, 조직 대량 생성 기능 개발
  • 목표 정보 상세보기 drawer 창 넓게/좁게 보기 기능 아이디어 제안
  • 목표 체크인 내역에 대한 반응하기 기능 추가 및 시험

PoC 중 대부분이 정식 제품에 성공적으로 반영되었습니다. 특히 대시보드 기능은 초기 체크인 현황에서 시작하여 승인 현황, 목표 등록 현황까지 확장되는 등 제품의 핵심 기능으로 발전했습니다. 내부 도구로 개발한 대량 생성 기능은 팀 전체의 QA 효율성을 크게 향상시켰으며, 작은 UX 개선 아이디어들도 사용자 만족도 향상에 기여했습니다.


빠른 프로토타이핑과 적극적인 제안 문화가 제품 개선에 얼마나 중요한 역할을 하는지 체감했습니다. 사용자 관점에서 발견한 작은 불편함도 놓치지 않고 즉시 해결책을 모색하는 세심함이 제품의 완성도를 크게 좌우한다는 것을 배웠습니다. 또한 개발자가 단순히 주어진 요구사항만 구현하는 것이 아니라, 능동적으로 문제를 발견하고 해결책을 제시할 때 팀 전체의 혁신 문화가 조성되고 최종적으로 더 나은 제품을 만들 수 있다는 점을 깨달았습니다.

문제
대규모 고객사(3,000명, 목표 30,000개) 도입을 앞두고 심각한 성능 저하가 예상되어 안정적인 서비스 제공을 위한 선제적인 최적화가 필수적이었습니다. 기존 시스템으로는 대규모 트래픽을 감당할 수 없음이 확인되어 구조 개선이 시급했고, 특히 위계 구조가 복잡한 목표 데이터의 조회 성능이 병목으로 작용하고 있었습니다. 서비스 중단 없이 기존 고객에게는 영향을 주지 않으면서도 대규모 환경에서 안정적으로 동작할 수 있는 시스템 구조가 요구되었습니다.
해결 과정
대규모 고객의 사용 환경과 유사한 상황이 필요하여 QA에 사용할 내부 툴을 개발했습니다.
  • 대규모 고객 환경과 유사한 테스트 데이터 생성을 위해 조직, 구성원을 대량으로 생성할 수 있는 내부 QA 도구 개발
  • 약 3,000명, 30,000개 목표 데이터를 생성하여 실제 대규모 환경 시뮬레이션 구성
  • 목표 조회 요청부터 데이터 조회, 응답 반환까지 전체 프로세스를 구간별로 나누어 병목 지점 측정 및 분석
  • 프로파일링 도구와 디버그 모드를 활용하여 성능 저하 구간을 체계적으로 식별하고 개선 우선순위 설정
  • 위계 데이터 처리용 anytree 패키지가 주요 병목임을 확인하고 근본적 해결을 위해 패키지 제거 및 최적화된 유틸 코드 새로 설계
결과

외부 플랫폼 연동 기능의 운영 복잡성과 ROI 평가
Slack 체크인 기능 개발 및 운영 경험
문제
크루 아이디어에서 시작된 고객 편의성 향상을 위한 Slack 체크인 기능 개발이 필요했습니다. PoC 과정을 거쳐 정식 배포하여 고객들이 Slack을 통해 더 편리하게 목표 체크인을 할 수 있도록 하는 것이 목표였습니다.
해결 과정
  • PoC 개발: 크루 아이디어를 바탕으로 프로토타입 구현 및 검증
  • Slack SDK 적용: Slack 다이얼로그 화면 처리를 백엔드에서 구현
  • 정식 배포: PoC 검증 후 프로덕션 환경에 정식 기능 출시
  • 문서화: 슬랙 SDK 사용법 및 장애 대응 가이드 작성
  • 고객 지원: 특정 회사들을 대상으로 기능 안내 및 설정 지원
결과
기능은 성공적으로 배포되었으나 운영상 심각한 문제점들이 드러났습니다. 장애 발생 시 고객의 Slack 환경 및 설정 파악이 어려워 원인 분석에 과도한 시간이 소비되었습니다. Slack SDK에 익숙하지 않은 다른 엔지니어들은 장애 대응에 어려움을 겪었고, 주요 사용자가 몇몇 특정 회사로 한정되어 기능의 ROI가 낮은 상태가 되었습니다.
